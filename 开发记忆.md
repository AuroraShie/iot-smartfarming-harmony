# 开发记忆

## 1. 项目当前结构

项目根目录当前主要分为两部分：

- `iottest`：ESP32 侧 IoT 工程，负责传感器采集、本地网关服务、执行器控制。
- `net/HarmoAgri`：HarmonyOS 前端工程，负责网关配置、设备展示、命令下发和实时状态消费。

当前约定是：

- 后续 IoT 代码修改只在 `D:\test111\iottest` 中进行。
- 不修改 HarmonyOS 侧 `net/HarmoAgri` 的网络协议适配逻辑，IoT 端去兼容它。

## 2. 鸿蒙端当前依赖的接口

HarmonyOS 前端当前依赖 ESP32 本地网关提供以下接口：

- `GET /gateway/status`
- `GET /telemetry/realtime`
- `GET /devices`
- `POST /devices/{id}/command`
- `GET /commands/{requestId}`
- `ws://<ip>:8080/ws`

WebSocket 主题约定：

- `telemetry.update`
- `device.status.changed`
- `command.result`

字段兼容要点：

- 实时遥测需要包含：`temperature`、`humidity`、`light`、`co2`、`eco2`、`tvoc`、`soilMoisture`、`soilTemperature`、`rainDetected`、`motionDetected`、`timestamp`
- `/devices` 里每个设备的 `telemetry` 目前走 `metricType/value/unit/timestamp` 结构
- `device.status.changed` 的 `data` 需要是数组
- 执行器设备 ID 保持：`pump-001`、`growlight-001`

## 3. IoT 端当前实现情况

传感器侧目前已经覆盖鸿蒙端实际消费的数据：

- DHT11：空气温度、空气湿度
- SGP30：TVOC、eCO2
- HW-390：土壤湿度
- DS18B20：土壤温度
- MH-RD：雨滴检测
- HC-SR501：人体红外检测
- 光敏传感器：光照状态

需要注意：

- 光照值当前仍然是数字量映射值，不是真实 lux，只是为了先适配鸿蒙端展示。
- 执行器现在已经拆成独立模块，不再只在网络层里维护软件状态。

## 4. 当前已经完成的结构调整

为了保留原有 IoT 项目风格，当前结构原则已经固定：

- 一种硬件一个独立 `.h/.cpp`
- `main.cpp` 只做初始化、调度、汇总和状态输出
- `config.h` 统一管理 GPIO 和系统参数

新增模块：

- `iottest/src/pump_controller.h`
- `iottest/src/pump_controller.cpp`
- `iottest/src/grow_light_controller.h`
- `iottest/src/grow_light_controller.cpp`

重构后的方向是：

- `main.cpp` 负责读取环境传感器、土壤传感器、状态类传感器
- `network_module` 负责本地 HTTP + WebSocket 网关
- 命令由 `main.cpp` 注册处理函数，最终驱动真实继电器模块

## 5. 硬件约定

当前默认硬件约定如下：

- `GPIO27`：水泵继电器
- `GPIO33`：补光继电器
- `GPIO2`：板载/手动指示灯，保留，不作为主补光输出
- 继电器触发方式：低电平触发

也就是说：

- `pump-001` 对应真实水泵继电器
- `growlight-001` 对应真实补光继电器
- 原先板载小灯继续作为本地状态指示，不再承担补光主执行器角色

## 8. 后续开发注意事项

后续继续开发时，建议保持以下规则：

- 不要再把执行器逻辑写回 `network_module.cpp` 内部做假状态维护
- 不要修改鸿蒙端现有网络接口解析，除非明确决定双端一起改
- 新增硬件时，继续沿用“一个模块一对 `.h/.cpp`”的结构
- 代码注释优先用中文，重点解释设计原因和硬件控制逻辑
- 任何涉及原项目文件的修改，都直接在 `D:\test111` 上进行
