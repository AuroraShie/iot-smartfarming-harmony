# 项目介绍与亮点分析

## 1. 项目简介

这是一个面向智慧农业场景的 HarmonyOS + IoT 联动项目，整体由三部分组成：

- IoT 设备端：`iottest`
- HarmonyOS 前端：`net/HarmoAgri`
- 本地联调工具：`net/HarmoAgri/tools/esp-mock-server.js`

当前项目的核心目标不是单独做一个传感器 demo，而是打通一条完整的数据闭环：

- 采集环境数据
- 统一建模
- 通过网关接口对外提供数据
- 被鸿蒙前端实时消费
- 支持设备控制命令回传

## 2. 项目结构

### 2.1 IoT 端

`iottest` 基于 ESP32，负责：

- 读取多种传感器数据
- 维护本地设备状态
- 暴露 HTTP + WebSocket 网关接口
- 接收执行器控制命令

### 2.2 HarmonyOS 端

`net/HarmoAgri` 负责：

- 网关配置保存
- HTTP 请求与 WebSocket 连接
- 页面状态管理与展示
- 设备控制与命令结果呈现

### 2.3 Mock 工具

`net/HarmoAgri/tools` 负责：

- 在真实 ESP 未就绪时提供模拟网关
- 降低前端开发阻塞
- 支持各种故障注入和状态演示

## 3. 当前架构亮点

## 3.1 本地网关模式更贴近真实设备

项目当前不是让 ESP32 把数据简单上传到某个固定服务器，而是让 ESP32 自己成为一个局域网网关节点。

这个设计的好处是：

- HarmonyOS 前端直接面向真实设备接口
- 更贴近真实农业场景中的边缘网关模型
- 后续替换为更强硬件时，接口层可以保持稳定

## 3.2 协议分层比较清晰

网关接口采用：

- HTTP：查询和控制
- WebSocket：实时推送

这种拆分非常适合设备联调：

- 静态状态和拉取型信息走 HTTP
- 变化快、需要实时反馈的信息走 WebSocket

对前端来说也更容易分层实现：

- `GatewayService` 处理请求
- `Repository / State` 处理数据流转
- 页面只关心展示和交互

## 3.3 传感器模块化程度高

`iottest` 的传感器代码是分模块封装的，不是把所有逻辑都堆在 `main.cpp` 里。

这带来几个优势：

- 单个传感器更容易替换
- 新增硬件时改动范围更小
- 故障隔离能力更强

对教学、演示、后续扩展都很友好。

## 3.4 既支持真实设备，又支持 mock 联调

这是这个项目很实用的一个点。

很多 IoT 项目一旦设备没连上，前端就完全卡住。但你这个项目保留了 Mock Server 路径，因此：

- 前端可以先独立开发
- 真实设备接入后再切换到真实网关
- 联调时可以快速对比“前端问题”和“设备问题”

这会显著降低协作成本。

## 3.5 命令闭环设计完整

项目不只是“能发命令”，而是支持完整闭环：

1. 前端发出控制命令
2. 网关返回 `accepted`
3. 设备状态更新
4. WebSocket 推送 `command.result`
5. 前端再确认最终执行结果

这个过程比单纯“POST 一下就结束”可靠得多，也更适合农业设备控制场景。

## 4. 业务亮点

从业务角度看，这个项目有三个很明显的亮点。

### 4.1 面向农业场景，不是泛 IoT 空壳

项目不是泛化到什么设备都能接，而是围绕农业温室/棚区这类实际场景展开：

- 温湿度
- 土壤湿度
- 光照
- CO2 / TVOC
- 雨滴
- 人体红外
- 灌溉水泵
- 补光灯

这使项目的展示完整度和说服力都更强。

### 4.2 既有监测，也有控制

很多项目只做到“看数据”，但你的项目已经进入“监测 + 控制”阶段。

这意味着它更像一个完整系统，而不是单方向采集 demo。

### 4.3 适合演示、课程、比赛和答辩

这个项目很适合做展示，因为它天然具备：

- 可视化页面
- 实时数据
- 硬件设备
- 执行器控制
- 联调工具

如果整理好文档和演示脚本，完整度会很高。

## 5. 当前最值得强调的技术价值

如果你后面要做汇报、答辩或 README 强化，最值得强调的是这三点：

- 端边协同：HarmonyOS 前端与 ESP32 网关在局域网内直接联动
- 协议统一：HTTP + WebSocket 的接口规范清晰，前后端边界明确
- 工程可扩展：真实设备、Mock Server、模块化传感器和执行器模型都已具备基础框架

## 6. 当前短板与后续提升方向

为了让介绍更真实，这里也建议把当前短板讲清楚。

### 6.1 设备控制仍偏演示态

目前执行器控制已经形成接口闭环，但硬件层安全保护、实际 GPIO 驱动和执行反馈还可以继续加强。

### 6.2 文档历史包袱较重

项目里原有文档里同时存在：

- 旧的“ESP 上传到 mock server”模式
- 新的“ESP 本地开网关”模式

如果不统一口径，新成员会比较容易混淆。

### 6.3 还可以进一步产品化

后续可以继续补：

- 历史数据
- 告警策略
- 自动控制规则
- 权限与鉴权
- 多设备发现与批量接入

## 7. 一句话总结

这是一个已经从“多传感器采集 demo”演进到“鸿蒙前端 + ESP32 网关 + 农业控制闭环”方向的项目，最大的价值在于真实联调链路已经成型，并且工程结构具备继续扩展的基础。
