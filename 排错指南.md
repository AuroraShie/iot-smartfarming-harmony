# 排错指南

## 1. 文档目标

这份文档用于统一处理当前项目最常见的问题，重点覆盖：

- ESP32 启动和烧录问题
- Wi-Fi 和局域网连接问题
- HTTP / WebSocket 接口问题
- HarmonyOS 前端联调问题
- Mock Server 联调问题

## 2. 先判断你在测哪种模式

排错前先确认你当前使用的是哪种模式：

### 模式 A：真实 ESP 网关

标志：

- ESP32 串口输出里出现 `Gateway is serving at http://<ip>:8080`
- 前端连的是 ESP32 的局域网 IP
- 端口通常是 `8080`

### 模式 B：Mock Server

标志：

- 你手动启动了 `node esp-mock-server.js`
- 控制台显示 `127.0.0.1:3000`
- 前端连的是电脑上的 Node 服务

很多排错失败，其实不是程序有问题，而是这两种模式混用了。

## 3. ESP32 相关问题

### 3.1 无法编译

现象：

- `pio run` 失败

优先检查：

- `PlatformIO` 是否安装正常
- `platformio.ini` 的板卡和串口是否正确
- 库依赖是否下载完成

建议操作：

```powershell
cd D:\test111\iottest
pio run
```

如果是串口问题，不影响编译本身；如果是依赖问题，通常会在日志里直接提示缺哪个库。

### 3.2 无法上传

现象：

- `pio run -t upload` 失败

优先检查：

- `platformio.ini` 里的 `upload_port`
- ESP32 是否真的连到了这个串口
- 串口是否被串口监视器占用

建议操作：

```powershell
pio device list
```

然后把 `upload_port` 改成实际串口。

### 3.3 串口无输出

优先检查：

- 波特率是否和 `SERIAL_BAUD_RATE` 一致
- 是否打开了正确串口
- 板子是否正常上电

建议操作：

```powershell
pio device monitor
```

## 4. Wi-Fi 连接问题

### 4.1 ESP32 无法连接 Wi-Fi

现象：

- 串口卡在连接阶段
- 输出 `Wi-Fi connection failed`

优先检查：

- `WIFI_SSID` 和 `WIFI_PASSWORD` 是否正确
- Wi-Fi 是否为 2.4G
- 信号是否过弱

建议处理：

- 确认路由器开启 2.4G
- 把板子移近路由器
- 先用手机热点快速验证

### 4.2 ESP32 与电脑不在同一网段

现象：

- ESP 启动正常
- 电脑访问不到 `http://<ESP_IP>:8080`

优先检查：

- 电脑和 ESP 是否连接同一 Wi-Fi
- IP 是否同网段，例如都在 `192.168.1.x`

建议操作：

```powershell
ping <ESP_IP>
```

如果 `ping` 不通，先不要怀疑接口代码，先解决网络层问题。

## 5. HTTP 接口问题

### 5.1 `/gateway/status` 访问失败

现象：

- 浏览器或 PowerShell 访问失败

优先检查：

- ESP 是否已打印 `Gateway is serving at ...:8080`
- 端口是不是 `8080`
- IP 是否写错

建议操作：

```powershell
Invoke-RestMethod http://<ESP_IP>:8080/gateway/status
```

### 5.2 `/devices` 返回不符合预期

现象：

- 鸿蒙前端能连通，但设备页展示异常

优先检查：

- 是否返回了 `pump-001`
- 是否返回了 `growlight-001`
- 传感器设备的 `telemetry` 是否带 `metricType/value/unit`

因为鸿蒙端现有 `GatewayService.ets` 是按这些字段结构解析的。

### 5.3 命令接口返回失败

现象：

- `POST /devices/{id}/command` 返回拒绝

优先检查：

- `deviceId` 是否正确
- `command` 是否受支持
- `requestId` 是否缺失

当前建议使用：

- `pump-001`：`TURN_ON`、`TURN_OFF`
- `growlight-001`：`TURN_ON`、`TURN_OFF`、`SET_LEVEL`

## 6. WebSocket 问题

### 6.1 HTTP 正常但 WebSocket 没消息

优先检查：

- 前端 `wsPath` 是否为 `/ws`
- 前端端口是否仍然写成 `3000`
- 是否真的连接的是 ESP IP 而不是旧 mock 地址

可先在浏览器测试：

```javascript
const ws = new WebSocket("ws://<ESP_IP>:8080/ws");
ws.onmessage = (e) => console.log(e.data);
```

### 6.2 能连上但控制后没有 `command.result`

优先检查：

- 控制命令是否真的被接口接收
- `requestId` 是否唯一
- `GET /commands/{requestId}` 能否查到结果

如果 HTTP 查询能查到结果，但 WebSocket 没收到，说明问题更偏向 WebSocket 推送链路。

## 7. HarmonyOS 前端联调问题

### 7.1 前端点“测试连接”失败

优先检查：

- `host` 是否填成 ESP32 实际 IP
- `port` 是否填成 `8080`
- `wsPath` 是否填成 `/ws`

如果你填的是 `10.0.2.2:3000`，那其实你走的是 mock server 模式，不是当前真实网关模式。

### 7.2 首页有状态，设备页为空

优先检查：

- `/devices` 是否正常返回数组
- 每个设备是否包含 `id/name/type/online/status/location/capabilities`

### 7.3 页面能显示，但控制后状态不更新

优先检查：

- `POST /devices/{id}/command` 是否返回 `accepted`
- `GET /commands/{requestId}` 是否能查到 `SUCCESS`
- WebSocket 是否收到 `device.status.changed`

## 8. Mock Server 相关问题

### 8.1 端口 3000 被占用

现象：

- 启动 `esp-mock-server.js` 报端口占用

建议处理：

```powershell
$env:ESP_MOCK_PORT=3001
node esp-mock-server.js
```

然后前端同步改成 `3001`。

### 8.2 模拟器无法访问电脑本机服务

优先检查：

- 模拟器模式下是否使用 `10.0.2.2`
- 电脑防火墙是否拦截了 Node 服务

## 9. 文档口径混淆问题

这是当前项目最容易踩的一个坑。

### 旧口径

旧文档中很多内容是：

- ESP32 主动向 mock server 上传数据
- 前端连 `3000`
- 模拟器大量使用 `10.0.2.2`

### 新口径

当前 `iottest` 已经改成：

- ESP32 自己开启本地网关
- 前端连 ESP32 局域网 IP
- 默认端口 `8080`
- WebSocket 路径 `/ws`

所以排错时一定先统一口径，不要混着看。

## 10. Git / Codex / VSCode 相关问题

### 10.1 Codex 里点“暂存”失败

常见原因：

- 当前 Codex 线程绑定的是临时 worktree
- 你想暂存的是 `D:\test111` 原目录文件
- UI 和实际仓库路径不一致

建议：

- 直接在 VSCode 打开 `D:\test111`
- 或直接在原目录执行 `git add`

### 10.2 改动明明有，但 VSCode 里看不到

优先检查：

- 你打开的是不是 `D:\test111`
- 不是 `C:\Users\Aurora\.codex\worktrees\...`

## 11. 最有效的排错顺序

推荐始终按这个顺序：

1. 先看 ESP 串口
2. 再测 HTTP
3. 再测 WebSocket
4. 最后看 HarmonyOS 页面

因为：

- 串口问题说明设备端都没起来
- HTTP 问题说明网关接口没通
- WebSocket 问题说明实时链路有问题
- 页面问题才是前端消费层问题

这个顺序最快，也最省时间。
